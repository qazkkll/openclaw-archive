## Round 1: 代码审计

### 1.1 日期格式不一致（严重）

**问题**: 系统中混用两种日期格式，导致比较和过滤可能失效。

| 位置 | 格式 | 示例 |
|------|------|------|
| `recommendation.date` | `%Y%m%d` | `20260622` |
| `recommendation.outcome_date` | `%Y-%m-%d` | `2026-06-23` |
| `recommendation_tracker.py:142` cutoff | `%Y%m%d` | `20251226` |
| `auto_scorer.py:185` today_str | `%Y%m%d` | `20260624` |
| `auto_scorer.py:218` outcome_date | `%Y-%m-%d` | `2026-06-24` |
| `compliance_tracker.py:75` snapshot date | `%Y-%m-%d` | `2026-06-24` |

**影响**: 
- `recommendation_tracker.py:143` 的180天cutoff用 `%Y%m%d` 字符串比较，但 `outcome_date` 是 `%Y-%m-%d` 格式——如果某天需要按outcome_date过滤，会比较失败
- `auto_scorer.py:194` 用 `rec_date >= today_str` 比较，如果rec_date是 `%Y-%m-%d` 格式（如 `2026-06-22`），而today_str是 `20260624`，字符串比较会错误地认为 `2026-06-22` > `20260624`（因为 `-` 的ASCII值 > 数字）

**修复建议**: 统一所有日期为 `%Y%m%d`，或统一为 `%Y-%m-%d`。至少在比较时先解析为datetime对象。

---

### 1.2 评分逻辑不匹配（严重）

**问题**: `recommendation_tracker.py` 和 `auto_scorer.py` 使用不同的评分阈值。

| 文件 | bullish涨阈值 | bullish跌阈值 | neutral阈值 |
|------|--------------|--------------|------------|
| `auto_scorer.py` (v3) | >2% = 1.0 | <-2% = 0.0 | <2% = 1.0 |
| skill文档 `recommendation-tracking` | >0.5% = 1.0 | <-0.5% = 0.0 | <1% = 1.0 |

**分析**: `auto_scorer.py` 的注释说"匹配模型训练标签（5天前瞻>2%）"，这是正确的。但skill文档中的阈值（0.5%）是旧版本的1天评分标准。skill文档与代码不一致，会误导Agent手动评分时使用错误阈值。

**影响**: 如果Agent参考skill文档手动评分（而非用auto_scorer.py），会使用0.5%阈值，与模型训练标签（2%）不匹配。

---

### 1.3 validate_recommendation() 规则校验缺陷

**问题 1: R006验证过于宽松**
```python
# recommendation_tracker.py:81
has_number = bool(re.search(r'\d+\.?\d*', reasoning))
```
这个正则匹配任何数字，包括日期（"2026"）、序号（"Top1"）、甚至"RSI=90"中的90。一个完全没有量化依据的推荐，如果reasoning里包含"2026年"或"Top3"，就会通过R006验证。

**问题 2: rules_applied自动回填掩盖问题**
```python
# recommendation_tracker.py:67
fixed['rules_applied'] = ['R004']  # 默认
```
当Agent没有填写rules_applied时，系统自动填入R004，这使得合规率统计失去意义——所有记录都有rules_applied，但很多是假的。

**问题 3: close_review验证有例外**
```python
# recommendation_tracker.py:75
if source == 'close_review' and action in ('buy', 'sell') and target != '大盘':
```
允许close_review对"大盘"产生买入/卖出推荐，但R005规则说"close_review只复盘不产生新推荐"，没有这个例外。

### 1.4 数据格式不一致

**scoring_detail类型混乱**:
- `auto_scorer.py` 写入dict: `{'base_price': ..., 'next_price': ..., 'horizon_days': 5}`
- 旧评分器写入string: `"entry=99.32, exit=95.48, ret=-3.87%"`

**actual_return类型混乱**:
- 有些记录是number: `actual_return: -3.87`
- 有些记录是string: `actual_return: "+0.50%"`
- `prediction_attribution.py:107` 尝试 `actual_return.replace('%', '')`，如果actual_return是number会报AttributeError

**score字段null值**:
- 多条status="scored"的记录score=null（如R7ce387d9 NXTC neutral, R5df4d181 HUN等）
- 这说明旧评分器写入了status=scored但没有写入score

### 4.4 信号分级系统（⚠️ 重要不一致）

**meta.json文档写的（绝对阈值）**：
```
🟢🟢 精品: pred_rank ≥ 0.90
🟢 强信号: pred_rank ≥ 0.80
🟡 观察: pred_rank ≥ 0.70
```

**评分脚本实际实现的（百分位阈值）**：
```python
# classify_signal_percentile()
p95 = np.percentile(all_scores, 95)
p90 = np.percentile(all_scores, 90)
p80 = np.percentile(all_scores, 80)
if score >= p95: return '🟢🟢', 'Top5%'
elif score >= p90: return '🟢', 'Top10%'
elif score >= p80: return '🟡', 'Top20%'
```

**关键区别**：
- meta.json的绝对阈值（0.90/0.80/0.70）来自Walk-Forward验证期的分布
- 生产脚本用的是**当日百分位**（Top5%/10%/20%），不看绝对值
- 这意味着：WF验证期和生产期的"🟢🟢"含义完全不同

**实际数据验证**：`output/v6_latest.json`中Top-15全部显示🟢🟢，但pred_rank值仅在0.5654~0.5938之间。这是因为这15只股票都在1847只股票的Top 5%（P95=0.5643）。绝对值远低于0.90，但百分位分级下确实是🟢🟢。

## Round 4-5 综合发现

### 🔴 严重问题

1. **信号文件路径不一致**：评分脚本输出到`output/`，但skill文档引用`signals/us/`。Agent无法自动读取最新信号。

2. **信号阈值文档与代码不一致**：meta.json写绝对阈值（0.90/0.80/0.70），生产脚本用百分位（Top5%/10%/20%）。如果Agent参考meta.json判断信号强度，会认为所有信号都不合格（实际pred_rank<0.60）。

3. **推荐质量极差**：10条有数据的推荐中仅1条命中（10%），bullish 0/7，系统性看多失败。

4. **VIX方向判断反复错误**：系统两次做空VIX均失败（Round 2: +12.5%反向，Round 5: -12.79%），说明VIX宏观判断模型有系统性缺陷。

### 🟡 需要关注

5. **分数压缩**：蓝盾V6的pred_rank有效范围仅0.48~0.58（跨度0.10），模型区分度有限。

6. **模型高自相关**：AC(1)>0.89，信号几乎不随时间变化。模型本质上是"选蓝筹"而非"选时机"。

7. **两个模型弱负相关**（r=-0.11）：组合使用有分散化效果，但也说明至少有一个模型方向性错误。

8. **parquet追踪文件仅10只股票**：`model_scores_us.parquet`仅追踪megacap，无法代表全市场评分分布。
